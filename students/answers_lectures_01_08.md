# Ответы на контрольные вопросы по лекциям 01-08

Формат: **полный вопрос -> подробный ответ**. Основная опора - лекции из `syllabus` 01-08; дополнительно использованы открытые источники по темам: NGINX reverse proxy, Docker best practices, Alistair Cockburn Hexagonal Architecture, Martin Fowler, Microsoft Learn по DDD и Google SRE.

## Лекция 01. Введение. Программное обеспечение сервера интернет-системы

### Вопрос 1. Что такое интернет-система? Из каких основных частей она состоит?

Интернет-система - это программный комплекс, доступный пользователям через сеть, чаще всего по HTTP/HTTPS. В лекции она рассматривается как система из клиентской части и серверной части. Клиентом может быть браузер, мобильное приложение или другой внешний потребитель API. Серверная часть включает reverse proxy, application server, базу данных, кэш, очереди сообщений, мониторинг и конфигурацию окружения.

Главная идея: интернет-система - это не только код приложения. Это набор взаимодействующих компонентов с разными зонами ответственности. Например, браузер отправляет запрос, reverse proxy принимает его, application server выполняет бизнес-логику, база данных хранит состояние, а очередь сообщений позволяет выполнять часть работы асинхронно.

### Вопрос 2. Для чего нужен reverse proxy? Назовите 3 задачи, которые он решает.

Reverse proxy - это входная точка серверной части, которая принимает внешние запросы и перенаправляет их внутренним приложениям. В лекции примеры таких инструментов: Nginx, Traefik, Caddy. Он закрывает приложение от прямого доступа из интернета и делает вход в систему управляемым.

Три главные задачи: TLS termination, маршрутизация и балансировка нагрузки. TLS termination означает, что HTTPS расшифровывается на уровне proxy. Маршрутизация позволяет отправлять `/api/*` в приложение, а `/static/*` - в файловый сервер. Балансировка распределяет запросы между несколькими экземплярами приложения. Дополнительно reverse proxy может выполнять rate limiting, добавлять заголовки, логировать входящий трафик и скрывать внутреннюю структуру системы.

### Вопрос 3. Что такое TLS termination и почему её часто выполняют на уровне reverse proxy?

TLS termination - это процесс завершения HTTPS-соединения на входной точке системы: reverse proxy принимает зашифрованный трафик, проверяет сертификат и расшифровывает запрос. Дальше внутри закрытой Docker-сети или внутренней сети запрос может передаваться как обычный HTTP.

Это часто делают на reverse proxy, потому что так проще централизовать работу с сертификатами, настройками HTTPS и безопасностью. Если каждое приложение само обслуживает TLS, то сертификаты, обновления и конфигурации приходится дублировать. Reverse proxy делает эту ответственность отдельной границей: безопасность входа находится у proxy, бизнес-логика - у application server.

### Вопрос 4. Объясните разницу между application server и reverse proxy.

Reverse proxy - сетевой посредник. Он принимает соединения, обрабатывает TLS, маршрутизирует запросы, балансирует нагрузку и может ограничивать частоту запросов. Он не должен содержать бизнес-правила предметной области.

Application server - это процесс приложения, где живут API-обработчики, сценарии использования, бизнес-логика, валидация и работа с данными. Например, в системе ПСО application server проверяет корректность заявки, определяет зону, создаёт запись в БД и инициирует уведомление группы. Поэтому reverse proxy отвечает за вход и инфраструктурные функции, а application server - за смысловую обработку запроса.

### Вопрос 5. Что такое граница ответственности? Приведите пример из рассмотренной архитектуры.

Граница ответственности - это линия разделения между компонентами, у каждого из которых своя задача и свой контракт взаимодействия. Компонент не должен знать внутренние детали другого компонента; он должен обращаться к нему через API, протокол, интерфейс или другой явный контракт.

Пример из лекции: граница между Nginx и приложением. Nginx знает, что запросы `/api/` нужно передать в `app:8000`, но он не знает бизнес-правила создания заявки. Приложение знает, как обработать заявку, но не управляет TLS-сертификатами и внешним маршрутом. Такая граница повышает сопровождаемость, тестируемость и безопасность.

### Вопрос 6. Почему зависимости должны быть направлены «внутрь» (к бизнес-логике)?

Зависимости направляют внутрь, чтобы бизнес-логика была самой устойчивой частью системы. Внешние детали - фреймворки, БД, HTTP, proxy, очереди - меняются чаще, чем правила предметной области. Если домен зависит от этих деталей, любое изменение инфраструктуры будет ломать ядро приложения.

Когда зависимости направлены к бизнес-логике, use cases и доменные классы не импортируют PostgreSQL, FastAPI или Nginx. Они работают через интерфейсы и порты. Это позволяет заменить PostgreSQL на другую БД, REST на gRPC или Nginx на Traefik без переписывания правил создания и обработки заявок.

### Вопрос 7. Что такое ADR? Зачем фиксировать архитектурные решения?

ADR, Architecture Decision Record, - это короткая запись архитектурного решения. В лекции ADR описан как дневник проекта: через время команда может понять, почему было принято именно такое решение, какие альтернативы рассматривались и какие последствия ожидались.

Фиксировать решения важно, потому что архитектура состоит из компромиссов. Например, выбор PostgreSQL вместо MySQL или TLS termination на Nginx влияет на безопасность, поддержку и эксплуатацию. Если решение не записать, команда потеряет контекст и может повторно спорить о том же самом или случайно нарушить важное ограничение.

### Вопрос 8. Почему «один контейнер - один процесс»? Что произойдёт при нарушении?

Принцип «один контейнер - один процесс» означает, что контейнер должен иметь одну главную ответственность и один понятный жизненный цикл. В Docker-подходе контейнер проще запускать, останавливать, проверять health check-ом и масштабировать, когда внутри работает один основной процесс.

Если в один контейнер положить Nginx, приложение и БД, появятся проблемы. Нельзя независимо перезапустить только приложение, сложно понять, какой процесс упал, логи смешиваются, health check становится неточным, масштабирование становится грубым. Поэтому в лекции пример строится как несколько сервисов: reverse proxy, app и db.

### Вопрос 9. Чем плох жёстко закодированный адрес базы данных в исходном коде?

Жёстко закодированный адрес БД привязывает приложение к конкретному окружению. В разработке адрес может быть `localhost`, в Docker Compose - `db`, на сервере - отдельный host или managed database. Если адрес находится в коде, для смены окружения приходится менять исходники и пересобирать приложение.

Правильнее передавать адрес через переменную окружения, например `DATABASE_URL`. Это отделяет конфигурацию от кода и соответствует идее границ: приложение знает, что ему нужен источник данных, но конкретный адрес задаёт окружение.

### Вопрос 10. Что означает `proxy_pass http://app:8000/` в конфигурации Nginx? Почему `app`, а не `localhost`?

`proxy_pass http://app:8000/` означает, что Nginx передаёт запросы во внутренний сервис `app` на порт 8000. В Docker Compose имя сервиса становится DNS-именем внутри общей сети, поэтому контейнер Nginx может обратиться к контейнеру приложения по имени `app`.

`localhost` внутри контейнера Nginx означает сам контейнер Nginx, а не приложение. Поэтому если написать `localhost:8000`, Nginx будет искать приложение внутри себя и, скорее всего, получит ошибку соединения. В Docker-сети нужно использовать имя сервиса.

### Вопрос 11. Какие архитектурные характеристики мы затронули в этой лекции? (Подсказка: безопасность, надёжность, сопровождаемость.)

В лекции затронуты безопасность, надёжность и сопровождаемость. Безопасность проявляется в TLS termination, reverse proxy как входной границе и ограничении прямого доступа к внутренним компонентам. Надёжность связана с health checks, отдельными контейнерами и корректным порядком старта сервисов.

Сопровождаемость появляется благодаря разделению ответственности: proxy отдельно, приложение отдельно, БД отдельно, конфигурация отдельно. Если нужно заменить Nginx, изменить БД или масштабировать приложение, изменения локализуются в одном компоненте.

### Вопрос 12. Как проверить, что приложение работает, не открывая браузер? Какой эндпоинт для этого предназначен?

Проверить приложение без браузера можно командой `curl`. В лекции для этого используется health endpoint, например `GET /health`. Он возвращает простой ответ вроде `{"status": "ok"}` и HTTP-статус `200 OK`.

Health endpoint нужен не только человеку. Его используют Docker, reverse proxy, orchestration-системы и мониторинг, чтобы автоматически понимать, живо ли приложение и можно ли направлять на него трафик.

### Вопрос 13. Назовите 3 сущности предметной области ПСО «Юго-Запад».

Примеры сущностей предметной области ПСО «Юго-Запад»: заявка, спасательная группа, операция. Также в лекциях встречаются зона, координаты, участник группы, диспетчер и координатор.

Важно понимать, что сущность предметной области - это не обязательно таблица БД. Это понятие из бизнеса. Например, заявка имеет жизненный цикл, приоритет, тип и статус, а спасательная группа имеет состав, доступность и возможность быть назначенной на заявку.

### Вопрос 14. В чём разница между use case и контроллером? Почему бизнес-логику не стоит помещать в контроллер?

Use case - это сценарий использования системы, то есть единица бизнес-логики приложения: создать заявку, назначить группу, закрыть операцию. Контроллер - это входной адаптер HTTP, который принимает запрос, разбирает параметры, вызывает use case и формирует ответ.

Если бизнес-логику поместить в контроллер, она станет зависеть от веб-фреймворка и HTTP. Тогда её сложнее тестировать без сервера, сложнее переиспользовать в gRPC, CLI или очереди сообщений. В архитектуре курса контроллер должен быть тонким, а бизнес-сценарий должен жить в application layer.

### Вопрос 15. Какие 2 ADR мы сформулировали? Какие альтернативы рассматривались?

В лекции сформулированы первые ADR: решение выполнять TLS termination на reverse proxy и решение выбрать СУБД для серверной системы, в примере - PostgreSQL. Эти ADR фиксируют не только выбор, но и причины выбора.

Для TLS альтернативами могли быть TLS в каждом application server или отсутствие отдельной TLS-границы. Для СУБД альтернативами могли быть MySQL, MS SQL, SQLite или NoSQL-хранилище. Выбор PostgreSQL связан со стабильностью, поддержкой транзакций, JSON, полнотекстового поиска и распространённостью в серверных системах.

## Лекция 02. Требования к интернет-системам и атрибуты качества. ADR

### Вопрос 1. В чём разница между функциональным и нефункциональным требованием? Приведите по одному примеру для ПСО «Юго-Запад».

Функциональное требование описывает, что система должна делать. Например: «диспетчер может создать заявку с типом, координатами и приоритетом». Это поведение системы с точки зрения пользователя или бизнес-процесса.

Нефункциональное требование описывает, насколько хорошо система должна это делать. Например: «заявка с приоритетом URGENT должна быть доставлена ближайшей свободной группе не позднее чем через 5 секунд при нагрузке 100 заявок в минуту в 99% случаев». Такое требование задаёт измеримые качества: скорость, надёжность, доступность, безопасность.

### Вопрос 2. Почему NFR определяют архитектуру в большей степени, чем FR? Приведите пример двух систем с одинаковыми FR, но разными NFR.

NFR часто сильнее влияют на архитектуру, потому что именно они заставляют добавлять кэш, очереди, репликацию, мониторинг, резервирование, таймауты и ретраи. Функционально две системы могут обе создавать заявки, но архитектурно быть совершенно разными.

Пример: учебная система ПСО и реальная диспетчерская система. В обеих можно создать заявку и назначить группу. Но учебной системе достаточно одного приложения и одной БД, а реальной системе нужны высокая доступность, наблюдаемость, отказоустойчивость, резервные каналы уведомлений и строгие SLO.

### Вопрос 3. Сформулируйте NFR для ПСО: «заявка с приоритетом URGENT должна быть доставлена ближайшей группе не позднее чем через ___». Укажите метрику, условие нагрузки и период.

Корректная формулировка должна быть измеримой. Например: «Заявка с приоритетом URGENT должна быть доставлена ближайшей свободной группе не позднее чем через 5 секунд после создания при нагрузке до 100 заявок в минуту; показатель выполняется для 99% таких заявок за календарный месяц».

Здесь есть метрика - время доставки до группы, условие нагрузки - до 100 заявок в минуту, период измерения - месяц, и целевой процент - 99%. Без этих элементов NFR будет слишком расплывчатым и непроверяемым.

### Вопрос 4. Назовите 7 ключевых атрибутов качества и объясните, чем «надёжность» отличается от «доступности».

Ключевые атрибуты качества: производительность, масштабируемость, доступность, надёжность, безопасность, сопровождаемость, наблюдаемость. В лекции также важны тестируемость и переносимость, но базовый набор именно такой.

Доступность показывает, какую долю времени система доступна пользователю, например 99.5% в месяц. Надёжность шире: система должна корректно работать, не терять данные, восстанавливаться после ошибок и сохранять ожидаемое поведение. Сервис может быть доступен, но ненадёжен, если он отвечает 200 OK, но создаёт некорректные заявки.

### Вопрос 5. Приведите пример компромисса между производительностью и надёжностью при кэшировании. Как TTL кэша влияет на оба атрибута?

Кэширование ускоряет ответы и снижает нагрузку на БД, то есть улучшает производительность. Но кэш может вернуть устаревшее значение, что ухудшает корректность и надёжность бизнес-процесса. В ПСО опасно слишком долго кэшировать статус группы: группа уже может быть занята, а система всё ещё считает её свободной.

TTL определяет срок жизни данных в кэше. Большой TTL даёт меньше обращений к БД и более быстрые ответы, но повышает риск устаревших данных. Маленький TTL делает данные свежее, но уменьшает выигрыш в производительности. Архитектурное решение должно фиксировать этот компромисс.

### Вопрос 6. Заполните сценарий качества (все 6 полей) для атрибута «масштабируемость» в ПСО «Юго-Запад».

Сценарий качества можно описать так. Источник стимула: резкий рост числа диспетчеров и заявок во время чрезвычайной ситуации. Стимул: система получает 500 заявок в минуту. Артефакт: API создания заявок и сервис назначения групп.

Окружение: рабочий режим, пик нагрузки, все основные сервисы доступны. Реакция: система масштабирует application server горизонтально, а очередь сглаживает пики обработки уведомлений. Мера реакции: p95 времени ответа API создания заявки не превышает 300 мс, а URGENT-заявки доставляются группе в пределах целевого SLO.

### Вопрос 7. Какие 5 обязательных разделов должны быть в ADR? Почему раздел «Альтернативы» критически важен?

В шаблоне лекции ADR включает контекст, затрагиваемые архитектурные характеристики, решение, альтернативы, последствия и проверку. Если нужно назвать пять обязательных, можно объединить характеристики с контекстом, но смысл должен сохраниться: почему возникла проблема, что выбрали, что ещё рассматривали, чем это обернётся и как проверим.

Раздел «Альтернативы» важен, потому что показывает осознанность выбора. Архитектура почти всегда является компромиссом. Если не записать альтернативы, через время будет непонятно, почему команда не выбрала другой вариант, например очередь вместо синхронного вызова или Redis вместо чтения из БД.

### Вопрос 8. Напишите ADR-005 для решения «хранение JWT-секрета в переменной окружения, а не в коде». Укажите контекст, альтернативы, последствия и проверку.

ADR-005: хранить JWT-секрет в переменной окружения. Контекст: JWT-секрет используется для подписи токенов; если он попадёт в Git, злоумышленник сможет подделывать токены. Решение: приложение читает секрет из переменной окружения `JWT_SECRET`, а при её отсутствии не стартует.

Альтернативы: хранить секрет в коде, хранить в конфигурационном файле в репозитории, использовать secret manager. Хранение в коде недопустимо из-за риска утечки. Файл лучше, но тоже опасен, если попадает в Git. Secret manager надёжнее, но сложнее для учебного проекта. Последствия: нужно настраивать окружение при запуске, зато секрет не хранится в исходниках. Проверка: поиск по репозиторию не находит реального секрета, тест запуска без `JWT_SECRET` падает, запуск с переменной проходит успешно.

### Вопрос 9. Чем SLI отличается от SLO? Что такое error budget и зачем он нужен?

SLI - это измеряемый показатель сервиса, например доля успешных HTTP-ответов, p95 latency или процент URGENT-заявок, доставленных группе вовремя. SLO - это целевое значение SLI, например «99.5% запросов успешны за месяц» или «p95 latency меньше 300 мс».

Error budget - допустимый запас нарушений SLO за период. Если SLO 99.5%, то 0.5% времени или запросов можно «потратить» на сбои. Google SRE использует error budget как инструмент баланса между скоростью изменений и стабильностью: если бюджет почти исчерпан, новые рискованные релизы лучше остановить.

### Вопрос 10. Что не так с NFR «система должна быть масштабируемой»? Перепишите его в измеримую форму.

Фраза «система должна быть масштабируемой» плоха тем, что её невозможно проверить. Неясно, что именно масштабируется, до какой нагрузки, с каким временем ответа и в какой период.

Измеримая форма: «При горизонтальном масштабировании application server до 4 экземпляров система должна обрабатывать 500 заявок в минуту, сохраняя p95 времени ответа `POST /api/requests` не выше 300 мс в течение 1 часа пиковой нагрузки». Такое требование можно протестировать нагрузочным тестом.

### Вопрос 11. Постройте цепочку NFR -> QAS -> ADR -> Проверка для атрибута «наблюдаемость» в ПСО.

NFR: «Все операции создания и назначения заявок должны логироваться с correlation id, а ключевые метрики должны быть доступны для мониторинга». QAS: источник - сбой при назначении группы; стимул - оператор сообщает, что URGENT-заявка не назначилась; артефакт - сервис dispatch; окружение - production; реакция - инженер по логам и метрикам находит причину; мера - не более 5 минут на первичную диагностику.

ADR: «Использовать структурированные JSON-логи, correlation id и метрики latency/error rate». Проверка: автотест или ручной сценарий создаёт заявку, затем проверяется наличие correlation id в логах, метрик в dashboard и трассировки пути запроса.

### Вопрос 12. Почему для ПСО надёжность важнее масштабируемости? В какой ситуации приоритеты поменяются?

Для ПСО надёжность важнее, потому что система связана со спасательными операциями. Если заявка потеряется, будет назначена неверная группа или статус окажется некорректным, последствия могут быть критичнее, чем медленная обработка большого числа запросов.

Приоритеты могут измениться во время массовой чрезвычайной ситуации, когда одновременно поступает резко больше заявок. Тогда масштабируемость становится почти такой же важной, как надёжность: система должна выдержать пик и при этом не нарушить ключевые инварианты.

### Вопрос 13. У вас есть error budget 0.5% в месяц. За первую неделю система нарушала SLO 2 часа. Стоит ли деплоить новую фичу? Обоснуйте.

0.5% месяца - это примерно 3.6 часа допустимого нарушения за 30 дней. Если за первую неделю уже потрачено 2 часа, израсходовано больше половины месячного error budget. Это сигнал, что система нестабильна.

Деплоить новую рискованную фичу не стоит, если она не исправляет причину нарушения SLO. Лучше направить работу на стабилизацию, исправление дефектов и снижение риска. Новую фичу можно выкатывать только аккуратно: через feature toggle, canary или после восстановления запаса надёжности.

### Вопрос 14. Заказчик просит 99.99% availability (≈ 4.38 мин простоя/мес). Какие архитектурные решения потребуются и во сколько раз вырастет стоимость по сравнению с 99.5%?

99.99% availability требует гораздо более сложной архитектуры: несколько экземпляров приложения, автоматический failover, резервирование БД, репликация, мониторинг, алерты, регулярные проверки восстановления, отказоустойчивый reverse proxy и часто несколько зон доступности. Для 99.5% можно допустить значительно больше простоя, поэтому архитектура и эксплуатация проще.

Точную стоимость нельзя вывести без проекта, но в лекции это показано как пример gold plating: повышение доступности на «четвёртую девятку» обычно увеличивает стоимость кратно. Платить приходится не только за серверы, но и за сложность эксплуатации, тестирование отказов, дежурства и мониторинг.

### Вопрос 15. Вернитесь к ADR-001 и ADR-002 из лекции 01. Какие разделы в них отсутствуют по сравнению с шаблоном из сегодняшней лекции? Дополните один из них.

В ранних ADR из лекции 01 обычно не хватает полного шаблона: явно прописанных альтернатив, затрагиваемых характеристик, последствий и проверки. В лекции 02 ADR становится более строгим документом, где решение связывается с качествами системы.

Дополнение к ADR о TLS termination: затрагиваемые характеристики - безопасность, сопровождаемость, переносимость. Альтернативы - TLS в каждом application server или отсутствие HTTPS на входе. Последствия - сертификаты централизованы, проще обновлять TLS, но reverse proxy становится критичной входной точкой. Проверка - запрос по HTTPS проходит, сертификат валиден, внутреннее приложение доступно только через proxy.

## Лекция 03. Принципы дизайна: SOLID, cohesion/coupling, границы модулей

### Вопрос 1. В чём разница между cohesion и coupling? Приведите примеры из ПСО «Юго-Запад».

Cohesion - это связность внутри модуля: насколько элементы модуля работают на одну ответственность. Высокая cohesion хороша. Например, модуль `dispatch` занимается созданием заявок и назначением групп, а не хранением пользователей или настройкой Nginx.

Coupling - это связанность между модулями: насколько один модуль зависит от деталей другого. Высокий coupling вреден. Плохой пример: `dispatch` напрямую импортирует внутренний класс PostgreSQL-репозитория из `operations`. Хороший пример: `dispatch` зависит только от публичного контракта или порта.

### Вопрос 2. Почему SRP важно применять на уровне модулей, а не только классов?

SRP говорит, что у элемента должна быть одна причина для изменения. На практике причины изменений часто проходят не по отдельным классам, а по модулям и фичам. Если модуль одновременно отвечает за заявки, уведомления, пользователей и отчёты, он будет меняться по множеству причин.

На уровне ПСО лучше отделить `dispatch`, `operations`, `resources`, `auth`. Тогда изменение правил назначения группы не затронет модуль аутентификации, а изменение Keycloak не заставит переписывать доменную модель заявок.

### Вопрос 3. Как принцип OCP помогает при добавлении нового типа заявки (`WATER_RESCUE`) в ПСО?

OCP означает, что код должен быть открыт для расширения, но закрыт для модификации. При добавлении `WATER_RESCUE` не стоит переписывать большой `if/elif` во всех местах. Лучше добавить новую стратегию, правило приоритизации или обработчик типа заявки.

Например, если расчёт приоритета сделан через интерфейс `PriorityPolicy`, то для водного спасения можно добавить `WaterRescuePriorityPolicy`. Старый код use case останется прежним, а новое поведение подключится через конфигурацию или фабрику.

### Вопрос 4. Как понять, что вы нарушили LSP при замене In-Memory репозитория на PostgreSQL?

LSP нарушен, если объект-наследник или реализация интерфейса не может заменить базовый контракт без изменения клиентского кода. В контексте репозитория это значит: use case должен одинаково работать с In-Memory и PostgreSQL реализацией.

Нарушение видно, если PostgreSQL-репозиторий возвращает ORM-объекты вместо доменных, иначе обрабатывает отсутствие записи, требует ручного commit внутри use case или бросает исключения, которых нет в контракте. Если тесты use case проходят с In-Memory, но требуют переписывания для PostgreSQL, контракт спроектирован плохо.

### Вопрос 5. Чем плох «толстый интерфейс» `RequestRepository` с 6 методами? Как ISP помогает?

Толстый интерфейс заставляет клиентов зависеть от методов, которые им не нужны. Например, use case создания заявки может использовать только `add`, но из-за общего интерфейса он формально зависит от поиска, удаления, отчётов и служебных операций.

ISP предлагает разделять интерфейсы по потребностям клиентов. Можно иметь отдельный порт для команд, отдельный для чтения или отдельные методы только там, где они действительно нужны. Это уменьшает связанность и упрощает тестовые реализации.

### Вопрос 6. Объясните правило зависимости (Dependency Rule) своими словами.

Правило зависимости говорит: зависимости в коде должны быть направлены к более важной и устойчивой политике, то есть к бизнес-логике. Внутренние слои не должны знать о внешних деталях.

В ПСО домен не должен импортировать FastAPI, SQLAlchemy, psycopg, Nginx или конкретный email-клиент. Наоборот, внешние адаптеры должны зависеть от портов и доменных объектов. Так бизнес-правила остаются независимыми от технических решений.

### Вопрос 7. Почему «ORM-модель как доменная сущность» - это антипаттерн?

ORM-модель описывает, как объект хранится в БД, а доменная сущность описывает бизнес-смысл и инварианты. Если объединить их, домен начнёт зависеть от схемы БД, сессии ORM, ленивой загрузки и технических ограничений.

Это нарушает DIP и усложняет тестирование. Доменная сущность `Request` должна уметь менять статус и проверять правила без подключения к БД. ORM-модель или SQL-строка должны преобразовываться в доменный объект через mapper в инфраструктуре.

### Вопрос 8. В чём преимущество feature-based структуры пакетов перед layer-based?

Layer-based структура группирует код по техническим слоям: `controllers`, `services`, `repositories`, `models`. При такой структуре одна бизнес-фича размазывается по всему проекту, и изменение заявки требует переходов между множеством папок.

Feature-based структура группирует код по бизнес-возможностям: `dispatch`, `operations`, `resources`. Внутри фичи могут быть свои application, domain, adapters. Это лучше поддерживает границы модулей и снижает случайную связанность между предметными областями.

### Вопрос 9. Что такое fitness-функция и как она защищает границы модулей?

Fitness-функция - это автоматическая проверка архитектурного свойства. Она работает как тест, но проверяет не конкретный ответ API, а структурное правило проекта.

Например, fitness-функция может проверить, что `domain/` не импортирует `infrastructure/`, что между `dispatch` и `operations` нет циклической зависимости, что контроллеры не обращаются напрямую к БД. Такая проверка помогает удерживать архитектурные решения не только в документах, но и в коде.

### Вопрос 10. Назовите 3 признака высокого coupling между модулями.

Первый признак - прямой импорт внутренней реализации другого модуля, например репозитория или ORM-модели. Второй - циклические зависимости, когда `dispatch` зависит от `operations`, а `operations` зависит от `dispatch`. Третий - изменения в одном модуле требуют правок в нескольких других.

Дополнительно о высоком coupling говорят общие глобальные структуры данных, знание внутренней схемы БД другого модуля и использование чужих приватных классов. Всё это делает систему хрупкой.

### Вопрос 11. Как DIP помогает тестировать use case без БД и без сети?

DIP заставляет use case зависеть от абстракций: портов репозитория, уведомлений, геолокации. Конкретные PostgreSQL, SMTP, HTTP-клиент или очередь подключаются снаружи.

В тесте вместо реальной БД подставляется In-Memory репозиторий, вместо email - fake notifier, вместо внешнего API - stub. Use case выполняется быстро и детерминированно, потому что бизнес-сценарий не требует сети и Docker.

### Вопрос 12. Нарисуйте граф зависимостей для модулей `dispatch`, `operations`, `resources` в ПСО. Есть ли циклы?

Текстово граф можно представить так: `dispatch` использует публичный контракт `operations` для проверки доступности группы и контракт `resources` для сведений о ресурсах. `operations` управляет группами и их состояниями. `resources` предоставляет информацию о ресурсах и не должен знать о внутренних сценариях `dispatch`.

Правильный граф должен быть ацикличным. Если `dispatch -> operations -> resources`, то обратной зависимости `resources -> dispatch` быть не должно. Цикл означает, что границы модулей выбраны плохо или общий контракт нужно вынести отдельно.

### Вопрос 13. Когда абстракция (порт/интерфейс) не нужна? Приведите пример.

Абстракция не нужна, если она не защищает границу и не даёт реальной заменяемости. Например, порт с одной реализацией, которая никогда не будет заменяться, может быть лишней сложностью.

Пример: простая функция форматирования номера заявки или локальный калькулятор без внешних зависимостей. Делать для неё интерфейс ради интерфейса не нужно. Абстракция оправдана, когда есть граница: БД, внешнее API, очередь, уведомления, файловая система.

### Вопрос 14. Как связаны принципы SOLID и атрибут качества «сопровождаемость» из лекции 02?

SOLID напрямую повышает сопровождаемость. SRP уменьшает причины изменений, OCP позволяет добавлять поведение без переписывания старого кода, LSP сохраняет заменяемость реализаций, ISP уменьшает лишние зависимости, DIP отделяет бизнес-логику от инфраструктуры.

Сопровождаемость означает, что систему можно менять с приемлемым риском и стоимостью. SOLID даёт практические правила, которые делают изменения локальными и уменьшают вероятность поломки соседних модулей.

## Лекция 04. Типовые решения построения интернет-систем

### Вопрос 1. В чём главное отличие Transaction Script от Domain Model? Когда каждый из них предпочтительнее?

Transaction Script организует бизнес-логику как набор процедурных сценариев: один сценарий обрабатывает одну операцию, например создание заявки. Это просто, понятно и быстро для CRUD-задач, где правил мало.

Domain Model переносит поведение в доменные объекты. Заявка сама контролирует переходы статусов, приоритет сам проверяет диапазон, группа сама защищает лимит участников. Domain Model предпочтительнее, когда появляются инварианты, сложные правила, жизненные циклы и риск дублирования логики.

### Вопрос 2. Какую роль играет Service Layer: содержит бизнес-правила или координирует шаги? Обоснуйте.

Service Layer должен координировать шаги use case: принять команду, загрузить агрегат, вызвать доменный метод, сохранить изменения, отправить уведомление, завершить транзакцию. Он задаёт сценарий приложения.

Бизнес-правила не должны разрастаться в Service Layer. Если правило относится к сущности или агрегату, оно должно быть в домене. Например, проверка допустимого перехода статуса заявки должна быть в `Request`, а не в application service.

### Вопрос 3. Почему «анемичная модель» считается антипаттерном? Приведите пример из ПСО «Юго-Запад».

Анемичная модель - это модель, где объекты содержат только данные, а вся логика вынесена в сервисы. Martin Fowler описывает её как антипаттерн, потому что она выглядит объектной, но фактически возвращает процедурный стиль с разорванной связью между данными и поведением.

Пример: `Request` содержит только `id`, `status`, `priority`, а сервисы снаружи решают, можно ли назначить группу, закрыть заявку или повысить приоритет. В богатой модели `request.assign_group(group_id)` и `request.close()` сами проверяют инварианты.

### Вопрос 4. В чём разница между Application Service и Domain Service? Дайте пример каждого.

Application Service реализует сценарий приложения. Он координирует доменные объекты и порты, но не содержит глубоких бизнес-правил. Пример: `CreateRequestUseCase` создаёт заявку, сохраняет её через репозиторий и инициирует уведомление.

Domain Service содержит доменную операцию, которая не принадлежит естественно одной сущности или Value Object. Пример: `NearestGroupSelectionService`, который по координатам заявки и состоянию групп выбирает ближайшую свободную группу.

### Вопрос 5. Как принцип DIP (лекция 03) связан с Service Layer? Почему use case не должен знать о PostgreSQL?

DIP требует, чтобы use case зависел от абстракций, а не от конкретных деталей. Service Layer должен работать с портом репозитория, например `RequestRepositoryPort`, а не с `psycopg`, SQL-запросами или ORM-сессией.

Если use case знает о PostgreSQL, он становится привязан к инфраструктуре. Его сложнее тестировать, сложнее перенести на другую БД и сложнее поддерживать. PostgreSQL должен находиться в adapter/infrastructure, а use case должен видеть только доменный контракт.

### Вопрос 6. Объясните, как MVC адаптируется к серверному API (REST). Что выступает в роли View?

В серверном REST-приложении Controller - это HTTP-обработчик или route, который принимает запрос и вызывает application service. Model - доменная модель или DTO, с которыми работает приложение.

View в классическом HTML-MVC отвечает за отображение. В REST API роль View выполняет представление данных в ответе: JSON DTO, response schema или view model. То есть пользователь не получает HTML-страницу, но получает структурированное представление ресурса.

### Вопрос 7. Почему в одной системе допустимо использовать разные шаблоны бизнес-логики для разных модулей?

Разные модули имеют разную сложность. Если часть системы является простым справочником, для неё достаточно Transaction Script. Если другая часть содержит сложные правила и инварианты, ей нужна Domain Model.

Архитектура не должна быть догмой. В ПСО заявки, операции и группы могут требовать богатой модели, а справочник типов заявок или регионов может оставаться простым CRUD. Это снижает избыточность и сохраняет понятность.

### Вопрос 8. Какие признаки указывают, что Transaction Script пора менять на Domain Model?

Признаки: в сценариях появляется много условий, правила дублируются, один и тот же статус проверяется в разных местах, изменения становятся опасными, а тесты приходится писать на длинные процедурные цепочки.

Например, если правила переходов заявки между статусами копируются в `create`, `assign`, `close`, `cancel`, значит поведение пора перенести в `Request`. Domain Model позволит держать инварианты рядом с данными.

### Вопрос 9. Какие проблемы возникнут, если контроллер напрямую вызывает репозиторий, минуя Service Layer?

Контроллер начнёт содержать бизнес-сценарий: загрузку данных, проверку правил, сохранение, транзакции. Это смешивает HTTP-слой с application logic и нарушает границы.

Проблемы: сложнее тестировать без HTTP, сложнее добавить другой входной адаптер, например gRPC или очередь, логика начинает дублироваться между контроллерами, транзакционная граница становится неявной. Service Layer нужен как единое место сценария использования.

### Вопрос 10. Как Table Module отличается от Active Record?

Table Module организует логику вокруг таблицы: один класс или модуль отвечает за операции над набором записей. Он не обязательно делает каждую строку объектом, который сам себя сохраняет.

Active Record совмещает данные строки, бизнес-методы и методы сохранения в одном объекте. В DDD это часто проблемно, потому что доменный объект начинает знать о БД. Table Module ближе к табличной логике, Active Record - к объекту, привязанному к записи.

### Вопрос 11. Приведите пример инварианта в классе `Request` из ПСО «Юго-Запад».

Инвариант - правило, которое всегда должно оставаться истинным для объекта или агрегата. Для `Request` пример: заявка не может быть закрыта, если она ещё не назначена группе или не находится в допустимом состоянии для закрытия.

Другие примеры: приоритет должен быть в допустимом диапазоне; нельзя назначить группу дважды без отмены старого назначения; URGENT-заявка должна иметь корректные координаты. Эти правила лучше защищать методами доменной модели.

### Вопрос 12. Что произойдёт, если Service Layer начнёт возвращать ORM-модели вместо DTO?

Если Service Layer возвращает ORM-модели, наружу протекает инфраструктура. Контроллеры и клиенты начинают зависеть от структуры таблиц, ORM-сессии, lazy loading и технических полей.

Это нарушает границу между application layer и infrastructure. Вместо этого Service Layer должен возвращать DTO или response model, специально подготовленные для API. Тогда внутреннюю схему БД можно менять без изменения внешнего контракта.

### Вопрос 13. Как fitness-функция (лекция 03) может защитить правило «контроллер не импортирует domain напрямую»?

Fitness-функция может анализировать импорты проекта. Например, тест проверяет, что файлы из `presentation/` или `controllers/` не импортируют `domain/`, а обращаются только к `application/`.

Если разработчик случайно добавит `from domain.request import Request` в контроллер, тест упадёт. Так архитектурное правило становится автоматической проверкой, а не только договорённостью в README.

### Вопрос 14. Сравните порог входа и стоимость сопровождения для Transaction Script и Domain Model на горизонте 2 лет.

Transaction Script имеет низкий порог входа: его легко написать и понять на старте. Для небольшой задачи он дешевле, потому что не требует проектировать сущности, Value Objects и агрегаты.

Но на горизонте 2 лет при росте правил стоимость сопровождения Transaction Script растёт: появляются дублирование и сложные условия. Domain Model требует большего порога входа, зато лучше удерживает сложность, потому что поведение и инварианты находятся рядом с данными.

## Лекция 05. Введение в предметно-ориентированное проектирование (DDD)

### Вопрос 1. Что такое DDD и в чём его главная ценность?

DDD, Domain-Driven Design, - подход к проектированию, где основой системы становится предметная область и модель бизнеса. Код должен говорить на языке экспертов, а не только на языке таблиц и HTTP-эндпоинтов.

Главная ценность DDD - согласование мышления бизнеса и разработки. Если в ПСО эксперты говорят «заявка», «операция», «группа», «назначение», эти понятия должны быть видны в коде и архитектуре. Это снижает разрыв между требованиями и реализацией.

### Вопрос 2. Объясните разницу между Core Domain, Supporting Subdomain и Generic Subdomain на примере ПСО «Юго-Запад».

Core Domain - ключевая область, ради которой существует система и где находится главное конкурентное или бизнес-значение. Для ПСО это управление заявками, операциями и назначением спасательных групп.

Supporting Subdomain поддерживает основную область, но не является центральной. Например, отчёты, уведомления или планирование смен. Generic Subdomain - типовая область, которую часто можно взять готовой: аутентификация, пользователи, роли, логирование.

### Вопрос 3. Почему единый язык (Ubiquitous Language) важен? Приведите пример «языковой мозоли».

Единый язык важен, потому что он устраняет разные трактовки одних и тех же слов. Разработчики, преподаватель, заказчик и пользователи должны одинаково понимать, что такое заявка, операция, группа и статус.

«Языковая мозоль» возникает, когда одно слово используется в разных смыслах. Например, «группа» может означать спасательную группу в operations, группу пользователей в Keycloak или группу заявок в отчёте. Если это не разделить контекстами, модель станет путаной.

### Вопрос 4. Что такое Bounded Context? Почему слово «Группа» означает разное в контекстах `dispatch` и `operations`?

Bounded Context - это граница, внутри которой модель и язык имеют конкретный смысл. Один и тот же термин в разных контекстах может означать разные вещи, и это нормально.

В `dispatch` группа может быть только ссылкой для назначения: `group_id`, доступность, расстояние до заявки. В `operations` группа - полноценная сущность с участниками, снаряжением, статусом, расписанием. Попытка сделать один универсальный класс `Group` приведёт к перегруженной модели.

### Вопрос 5. Какие паттерны интеграции контекстов вы знаете? Когда уместен Anti-Corruption Layer?

Паттерны интеграции контекстов: Partnership, Customer/Supplier, Conformist, Shared Kernel, Published Language, Open Host Service, Anti-Corruption Layer. В лекциях особенно важны Published Language и ACL.

Anti-Corruption Layer уместен, когда внешний контекст или сервис имеет чужую модель, которую нельзя напрямую пускать в наш домен. Например, Keycloak говорит языком realm, role, group, token, а ПСО говорит языком диспетчера, координатора и спасательной группы. ACL переводит внешнюю модель во внутреннюю.

### Вопрос 6. Постройте карту контекстов для системы, в которой есть `orders`, `payments`, `shipping`.

Карта может быть такой: `orders` - контекст оформления заказа и его жизненного цикла. `payments` - контекст оплаты, который принимает команду или событие о заказе и возвращает статус платежа. `shipping` - контекст доставки, который начинает работу после успешной оплаты.

Интеграция: `orders` публикует язык событий, например `OrderPlaced`, `OrderPaid`. `payments` сообщает `PaymentSucceeded` или `PaymentFailed`. `shipping` подписывается на успешную оплату и создаёт отправление. Между контекстами нужны явные контракты, а не прямой доступ к внутренним таблицам.

### Вопрос 7. Как связаны Bounded Context и feature-based packaging (лекция 03)?

Bounded Context задаёт бизнес-границу модели, а feature-based packaging помогает выразить эту границу в структуре кода. Вместо технических папок на весь проект появляются модули по смыслу: `dispatch`, `operations`, `resources`.

Внутри каждого контекста могут быть свои domain, application и adapters. Так структура проекта поддерживает DDD: код, язык и границы изменений совпадают с предметной областью.

### Вопрос 8. Почему «один класс Group на 40 полей» - антипаттерн?

Один огромный класс `Group` на 40 полей, скорее всего, смешивает разные контексты. В нём окажутся поля для спасательной группы, группы пользователей, отчётной группы, расписания, оборудования и прав доступа.

Такой класс трудно понимать, тестировать и изменять. Он нарушает SRP и Bounded Context. Лучше иметь разные модели: `GroupRef` в dispatch, `RescueGroup` в operations, `UserGroup` в auth.

### Вопрос 9. Когда тактическое DDD излишне? Приведите пример.

Тактическое DDD излишне, когда предметная область простая и почти не содержит бизнес-правил. Если модуль является обычным CRUD-справочником, полноценные агрегаты, доменные события и сложные Value Objects могут быть лишней стоимостью.

Пример: справочник типов заявок или список регионов, если там нет сложного жизненного цикла. Для такого модуля достаточно простого application service или Transaction Script.

### Вопрос 10. Что такое контракт интеграции контекстов? Чем Published Language отличается от внутренней модели?

Контракт интеграции - это явное соглашение между контекстами: формат сообщений, API, события, DTO, коды ошибок. Он определяет, как контексты взаимодействуют, не раскрывая внутренние модели.

Published Language - опубликованный стабильный язык интеграции. Он отличается от внутренней модели тем, что предназначен для внешних потребителей и должен быть стабильнее. Внутренний класс можно переименовать или изменить, но публичный контракт нужно менять осторожно.

### Вопрос 11. Как ACL защищает контекст от изменений внешнего сервиса (Keycloak)?

ACL находится между нашим контекстом и Keycloak. Он преобразует внешние понятия Keycloak в понятия нашего домена. Например, role или group из Keycloak превращается в `UserRole` или `DispatcherPermissions` внутри ПСО.

Если Keycloak изменит формат токена или структуру групп, изменения локализуются в ACL. Домен dispatch не должен знать, как именно Keycloak хранит роли. Это защищает модель от загрязнения чужими терминами.

### Вопрос 12. Как связаны стратегическое DDD и ADR? Приведите пример ADR, который фиксирует решение о контекстах.

Стратегическое DDD принимает архитектурные решения о поддоменах, bounded contexts и интеграции между ними. ADR нужен, чтобы эти решения зафиксировать: почему выделили такие контексты, какие альтернативы были и какие последствия.

Пример ADR: «ADR-010: разделить систему ПСО на контексты `dispatch` и `operations`». Контекст: слово «группа» имеет разные смыслы. Решение: в `dispatch` хранить только `GroupRef`, в `operations` - полноценный агрегат `Group`. Последствия: нужна интеграция через Published Language, зато модели не смешиваются.

### Вопрос 13. В чём разница между стратегическим и тактическим DDD?

Стратегическое DDD отвечает на вопросы крупной архитектуры: какие есть поддомены, где core domain, какие bounded contexts нужны, как они интегрируются. Это уровень карты системы.

Тактическое DDD работает внутри контекста. Оно говорит, как моделировать Entity, Value Object, Aggregate, Repository, Domain Service. То есть стратегическое DDD определяет границы, а тактическое - внутреннее устройство модели.

### Вопрос 14. Как определить, что два понятия принадлежат разным контекстам?

Понятия принадлежат разным контекстам, если у них разные правила, разные владельцы данных, разные жизненные циклы или разные смыслы в языке экспертов. Также признак - изменения одного понятия не должны автоматически менять другое.

Например, «группа» в auth живёт как набор пользователей и прав, а «группа» в operations - как спасательное подразделение с участниками и доступностью. Несмотря на одинаковое слово, это разные модели и разные bounded contexts.

## Лекция 06. Архитектура: гексагональная, чистая, порты и адаптеры

### Вопрос 1. Чем «большой комок грязи» отличается от послойной архитектуры? Почему послойная тоже может быть проблемной?

Big Ball of Mud - это система без ясных границ, где всё зависит от всего: HTTP, SQL, бизнес-правила и формат ответа смешаны в одном месте. Такая система быстро становится хрупкой.

Послойная архитектура лучше, потому что разделяет presentation, application, domain, infrastructure. Но она тоже может быть проблемной, если зависимости идут сверху вниз до БД, а домен фактически зависит от ORM. Тогда слой БД начинает диктовать модель, и правило зависимости нарушается.

### Вопрос 2. Объясните разницу между driving port и driven port. Приведите примеры из ПСО «Юго-Запад».

Driving port - входной порт, через который внешний мир запускает приложение. Пример: порт `CreateRequestUseCase` или интерфейс команды создания заявки. Его может вызывать REST controller, CLI или consumer очереди.

Driven port - исходящий порт, через который приложение обращается к внешним ресурсам. Примеры: `RequestRepositoryPort`, `NotificationPort`, `GeoLocationPort`. Use case вызывает эти порты, а конкретные адаптеры работают с PostgreSQL, email, push или внешним API.

### Вопрос 3. Сформулируйте Dependency Rule своими словами. Какой слой «не знает» о каком?

Dependency Rule: внутренние слои не знают о внешних. Domain не знает об application, adapters, FastAPI, PostgreSQL. Application может знать домен и порты, но не должен знать конкретные адаптеры.

Внешние слои знают о внутренних: REST controller знает, какой use case вызвать; PostgreSQL adapter знает, как собрать доменный объект. Но доменный объект не должен знать, кто его сохранил и через какой HTTP-роут он был создан.

### Вопрос 4. Чем гексагональная архитектура отличается от Clean Architecture? В чём они одинаковы?

Гексагональная архитектура, описанная Cockburn, говорит о приложении как о ядре, окружённом портами и адаптерами. Входящие адаптеры запускают use cases, исходящие адаптеры реализуют доступ к внешним системам.

Clean Architecture описывает похожую идею через концентрические круги: entities, use cases, interface adapters, frameworks. Общее у них - бизнес-логика в центре, внешние детали снаружи, зависимости направлены внутрь. Разница больше в языке описания и акцентах.

### Вопрос 5. Почему REST-эндпоинт `POST /requests/{id}/assign` лучше, чем `PATCH /requests/{id}` со значением `groupId`?

`POST /requests/{id}/assign` выражает бизнес-команду «назначить группу». Он показывает смысл операции и позволяет явно реализовать правила назначения: проверить статус заявки, доступность группы, координаты и ограничения.

`PATCH /requests/{id}` с `groupId` выглядит как техническое изменение поля. Такой API превращает доменную операцию в CRUD и скрывает бизнес-смысл. В DDD API должен отражать язык предметной области, особенно для команд с правилами.

### Вопрос 6. Как Composition Root связывает порты и адаптеры? Зачем он нужен?

Composition Root - место, где приложение собирает зависимости. Там создаются конкретные адаптеры: PostgreSQL repository, email notifier, HTTP clients. Затем они передаются в use cases через интерфейсы портов.

Он нужен, чтобы домен и use cases не создавали инфраструктуру сами. Например, `CreateRequestUseCase` принимает `RequestRepositoryPort`, а Composition Root решает, будет это `PostgresRequestRepository` в production или `InMemoryRequestRepository` в тестах.

### Вопрос 7. Как написать unit-тест для use case без БД и без HTTP?

Нужно тестировать use case напрямую, без контроллера. Вместо реального репозитория подставить In-Memory или fake-репозиторий, вместо уведомлений - fake notifier, вместо внешнего API - stub.

Тест создаёт команду, вызывает `use_case.execute(command)`, а затем проверяет состояние fake-репозитория и вызовы fake-уведомлений. Так проверяется бизнес-сценарий, а не работа FastAPI, Docker или PostgreSQL.

### Вопрос 8. Что такое API-first подход? Как он связан с Published Language (лекция 05)?

API-first означает, что внешний контракт проектируется до или вместе с реализацией, а не случайно получается из внутренних классов. Сначала определяется, какие команды, ответы, ошибки и схемы видит потребитель.

Это связано с Published Language, потому что API становится опубликованным языком контекста. Он должен быть понятен другим контекстам и стабилен. Внутреннюю модель можно менять, но опубликованный контракт требует аккуратной эволюции.

### Вопрос 9. Почему Domain Model не должна наследоваться от ORM-базы (например, `Base`)?

Если доменная модель наследуется от ORM `Base`, она становится частью инфраструктуры. В домене появляются технические детали: таблицы, колонки, сессии, lazy loading, ограничения ORM.

Это нарушает Dependency Rule. Домен должен быть чистым и тестироваться без БД. ORM-модель или mapper должны находиться в infrastructure/adapters и переводить данные между таблицей и доменным объектом.

### Вопрос 10. Как fitness-функция может проверить, что `domain/` не зависит от `adapters/`?

Fitness-функция может анализировать импорт-граф проекта. Она проверяет, что файлы в `domain/` не импортируют модули `adapters/`, `infrastructure/`, `fastapi`, `sqlalchemy`, `psycopg` и другие внешние детали.

Технически это можно сделать через import-linter, pytest-проверку или статический анализ. Если кто-то добавит запрещённый импорт, тест упадёт и покажет нарушение архитектурной границы.

### Вопрос 11. В каких случаях гексагональная архитектура избыточна?

Она избыточна для маленьких прототипов, одноразовых скриптов и простых CRUD-модулей без сложной бизнес-логики. Если нет потребности заменять БД, тестировать сценарии отдельно или поддерживать долгую эволюцию домена, слои портов и адаптеров могут быть лишней сложностью.

Но для ПСО с заявками, группами, внешними уведомлениями и требованиями к надёжности гексагональная архитектура полезна. Она защищает бизнес-правила от инфраструктурных изменений.

### Вопрос 12. Как гексагональная архитектура помогает заменить PostgreSQL на MongoDB без изменения бизнес-логики?

Use case зависит от `RequestRepositoryPort`, а не от PostgreSQL. PostgreSQL-адаптер реализует этот порт через SQL и таблицы. Если нужно перейти на MongoDB, создаётся новый `MongoRequestRepository`, который реализует тот же порт.

Домен и application layer не меняются, потому что их контракт прежний: `add`, `get_by_id`, `save`, `remove` или другие доменные методы. Меняется только внешний адаптер и маппинг.

### Вопрос 13. Сколько адаптеров может быть у одного порта? Приведите пример.

У одного порта может быть сколько угодно адаптеров, если они соблюдают один контракт. Это одна из главных идей ports & adapters.

Пример: `RequestRepositoryPort` может иметь `InMemoryRequestRepository` для unit-тестов, `PostgresRequestRepository` для production, `MongoRequestRepository` для эксперимента и `FakeRequestRepository` для отдельных тестов. Use case не должен отличать эти реализации.

### Вопрос 14. Как связаны Bounded Context (лекция 05) и гексагон?

Bounded Context задаёт бизнес-границу модели, а гексагональная архитектура помогает организовать код внутри этой границы. Каждый контекст можно представить как отдельный гексагон: внутри домен и use cases, снаружи входящие и исходящие адаптеры.

Например, `dispatch` имеет свои команды, модель заявки и порты к группам/уведомлениям. `operations` имеет свою модель групп. Они не лезут внутрь друг друга, а взаимодействуют через опубликованные контракты.

## Лекция 07. Доменные модели: сущности, объекты-значения, агрегаты

### Вопрос 1. Чем Entity отличается от Value Object? Приведите по два примера из ПСО «Юго-Запад».

Entity имеет идентичность. Две сущности могут иметь одинаковые поля, но оставаться разными объектами, если у них разные ID. Примеры: `Request` и `Group`, потому что заявка и группа имеют жизненный цикл и идентификатор.

Value Object не имеет самостоятельной идентичности и определяется значениями. Если значения равны, объекты считаются равными. Примеры: `Coordinates` и `Priority`. Координаты важны как пара latitude/longitude, а приоритет - как значение уровня срочности.

### Вопрос 2. Почему `Coordinates` - Value Object, а `Request` - Entity?

`Coordinates` - Value Object, потому что у них нет собственного жизненного цикла и ID. Важны только значения широты и долготы. Две координаты с одинаковыми lat/lon означают одно и то же место.

`Request` - Entity, потому что заявка сохраняет идентичность при изменении статуса, приоритета или назначенной группы. Заявка может менять поля, но оставаться той же самой заявкой по ID.

### Вопрос 3. Что такое Aggregate Root? Зачем нужен единственный «вход» в агрегат?

Aggregate Root - корневая сущность агрегата, через которую внешний код взаимодействует со всем агрегатом. Агрегат - группа объектов, для которых нужно вместе защищать инварианты.

Единственный вход нужен, чтобы никто не менял внутренние объекты в обход правил. Например, участника группы нельзя просто добавить в список напрямую; нужно вызвать `group.add_member(member)`, чтобы проверить лимит, дубли и состояние группы.

### Вопрос 4. Объясните правило «один агрегат - одна транзакция». Что делать, если нужно обновить два агрегата?

Правило означает, что в одной транзакции безопасно и желательно изменять один агрегат, потому что агрегат является границей инвариантов. Так модель остаётся проще, а транзакции не разрастаются.

Если нужно обновить два агрегата, чаще используют доменное событие и eventual consistency. Например, заявка назначена группе - агрегат `Request` сохраняет `group_id`, а затем событие сообщает `operations`, что группа должна изменить состояние. Согласованность между агрегатами достигается не обязательно мгновенно.

### Вопрос 5. Почему ссылка между агрегатами должна быть по ID, а не по объекту?

Ссылка по объекту связывает агрегаты слишком сильно. Если `Request` хранит объект `Group`, то загрузка заявки может потянуть всю группу, участников, расписание и другие данные. Это раздувает границы транзакции.

Ссылка по ID делает связь явной и лёгкой. `Request` хранит `group_id`, а если нужно получить подробности группы, приложение обращается к соответствующему репозиторию или контексту. Это поддерживает слабую связанность и правило границ агрегатов.

### Вопрос 6. Чем анемичная модель хуже богатой? Приведите пример миграции от анемичной к богатой.

Анемичная модель хуже тем, что данные и поведение разделены. Объекты становятся контейнерами полей, а правила расползаются по сервисам. Это приводит к дублированию условий и риску обойти инварианты.

Миграция: было `EscalationService.escalate(request)`, который снаружи менял `request.priority`. Стало `request.escalate()`, где сама заявка проверяет, можно ли повысить приоритет, меняет значение через Value Object `Priority` и сохраняет корректное состояние.

### Вопрос 7. Как Value Object `Priority` реализует операцию `escalated()` без мутации?

Value Object должен быть неизменяемым. Поэтому `Priority.escalated()` не меняет поле текущего объекта, а возвращает новый `Priority` с повышенным значением.

Например, был `Priority(2)`, вызов `escalated()` возвращает `Priority(3)`. Старый объект остаётся прежним. Это делает поведение предсказуемым и безопасным, особенно если объект используется в нескольких местах.

### Вопрос 8. Почему `frozen=True` важен для Value Objects?

`frozen=True` в dataclass запрещает менять поля после создания объекта. Для Value Object это важно, потому что равенство и корректность зависят от набора значений.

Если координаты или приоритет можно изменить после создания, объект перестаёт быть надёжным значением. Неизменяемость упрощает тестирование, сравнение, использование в словарях и защиту инвариантов.

### Вопрос 9. Назовите 4 правила проектирования агрегатов по Вернону.

Основные правила: проектировать маленькие агрегаты; защищать инварианты внутри агрегата; ссылаться на другие агрегаты по ID, а не по объектам; изменять один агрегат в одной транзакции.

Также важное следствие: между агрегатами стоит использовать eventual consistency и доменные события. Агрегат не должен становиться огромным объектным графом, иначе он теряет смысл как управляемая граница.

### Вопрос 10. Как связаны Aggregate и Bounded Context?

Aggregate существует внутри Bounded Context и подчиняется языку этого контекста. Его правила, поля и методы имеют смысл только в этой модели.

Например, `Group` в operations может быть агрегатом с участниками и лимитом. В dispatch вместо такого агрегата может быть только `GroupRef`. Это разные модели в разных контекстах, даже если слово одинаковое.

### Вопрос 11. Как выбрать между UUID и auto-increment для идентификации сущностей?

UUID удобен, когда идентификатор нужен до сохранения в БД, когда есть распределённые сервисы или когда доменное событие должно содержать ID ещё до commit. Он меньше привязан к конкретной СУБД.

Auto-increment проще и хорошо подходит для монолитной БД, где ID выдаёт сама таблица. Но он сложнее в распределённых сценариях и привязывает создание идентичности к сохранению записи. В DDD часто выбирают UUID для агрегатов.

### Вопрос 12. Что произойдёт, если внешний код напрямую модифицирует `group.members.append(member)` в обход `group.add_member()`?

Внешний код обойдёт инварианты агрегата. Он сможет добавить 11-го участника при лимите 10, добавить дубль, добавить участника в закрытую или недоступную группу.

Поэтому внутренние коллекции агрегата нельзя отдавать наружу для прямой модификации. Нужно предоставлять методы вроде `add_member()`, которые проверяют правила и сохраняют корректное состояние.

### Вопрос 13. Как протестировать инвариант «max 10 участников в группе» без БД?

Это unit-тест доменной модели. Нужно создать объект `Group`, добавить 10 участников через `add_member()`, затем попытаться добавить 11-го и проверить, что метод бросает доменное исключение.

БД не нужна, потому что инвариант принадлежит агрегату, а не PostgreSQL. Такой тест быстрый и точно показывает, что правило защищено в домене.

### Вопрос 14. Как агрегаты связаны с границами транзакции в PostgreSQL?

Агрегат задаёт логическую границу транзакции: всё, что нужно изменить для сохранения его инвариантов, сохраняется атомарно. В PostgreSQL это может быть одна транзакция, которая обновляет таблицу агрегата и связанные внутренние записи.

Но транзакция не должна без необходимости охватывать несколько агрегатов. Если `Request` и `Group` являются разными агрегатами, лучше связать их через ID и события, а не пытаться всегда обновлять оба в одной большой транзакции.

## Лекция 08. Репозитории: инверсия зависимостей, коллекция доменных объектов, транзакции

### Вопрос 1. Почему интерфейс `RequestRepositoryPort` определяется в `domain/`, а не в `infrastructure/`?

Порт репозитория определяется там, где он нужен бизнес-логике. Домен или application layer задаёт контракт: какие операции нужны для работы с агрегатом `Request`. Infrastructure только реализует этот контракт.

Если интерфейс лежит в infrastructure, домен начинает зависеть от внешнего слоя, что нарушает DIP. Правильная зависимость обратная: PostgreSQL-адаптер зависит от доменного порта и реализует его.

### Вопрос 2. Чем Repository отличается от DAO?

Repository представляет коллекцию доменных агрегатов. Его методы работают на языке домена: `add(request)`, `get_by_id(id)`, `save(request)`. Он скрывает, где и как хранятся данные.

DAO обычно ниже уровнем и работает с записями, строками таблиц или SQL-операциями. DAO ближе к структуре хранения, а Repository - к доменной модели. Поэтому в DDD Repository должен возвращать агрегаты, а не сырые строки БД.

### Вопрос 3. Почему Repository работает с Aggregate Root, а не с любой Entity?

Aggregate Root - единственная внешняя точка доступа к агрегату. Если репозиторий позволит сохранять внутренние Entity отдельно, внешний код сможет нарушить инварианты.

Например, не нужен отдельный `MemberRepository`, если `Member` является внутренней частью агрегата `Group`. Нужно загружать и сохранять `Group`, а добавление участника выполнять через `group.add_member()`.

### Вопрос 4. Объясните роль `_row_to_entity` в PostgreSQL-адаптере. К какому паттерну Фаулера это относится?

`_row_to_entity` переводит строку БД в доменный объект. Например, значения `lat` и `lon` собираются в `Coordinates`, строка статуса превращается в enum, а UUID становится идентификатором `Request`.

Это относится к паттерну Data Mapper у Фаулера. Data Mapper отделяет доменную модель от структуры хранения. Благодаря этому домен не знает о таблицах, колонках и SQL.

### Вопрос 5. Как In-Memory репозиторий помогает тестированию? Приведите пример теста.

In-Memory репозиторий реализует тот же порт, что и PostgreSQL-репозиторий, но хранит агрегаты в словаре. Он позволяет тестировать use cases без Docker, миграций, сетевых задержек и реальной БД.

Пример теста: создать `InMemoryRequestRepository`, передать его в `CreateRequestUseCase`, выполнить `execute(command)`, затем проверить, что `repo.get_by_id(result.id)` возвращает созданную заявку с нужным типом и приоритетом.

### Вопрос 6. Что такое Unit of Work? Зачем он нужен, если Repository уже умеет `save()`?

Unit of Work управляет транзакционной границей use case. Он отслеживает изменения, открывает транзакцию, выполняет commit при успехе и rollback при ошибке.

Repository умеет сохранить отдельный агрегат, но не всегда должен сам решать, когда фиксировать транзакцию. Если use case делает несколько операций, commit в каждом методе репозитория разрушит атомарность. UoW переносит решение о commit на уровень сценария.

### Вопрос 7. Что произойдёт, если use case бросит исключение внутри `with uow:`?

Если use case бросит исключение внутри `with uow:`, метод `__exit__` у Unit of Work должен выполнить rollback. Это означает, что изменения не фиксируются в БД.

Такой подход делает поведение безопасным: если на середине сценария не удалось отправить событие или сохранить агрегат, система не оставляет частично применённые изменения. В тестовом UoW можно проверить, что `rolled_back is True`.

### Вопрос 8. Как Composition Root связывает порт с адаптером? Покажите на примере.

Composition Root создаёт конкретную реализацию и передаёт её туда, где ожидается порт. Например: `repo = PostgresRequestRepository(connection)`, затем `use_case = CreateRequestUseCase(repo)`. Тип use case ожидает `RequestRepositoryPort`, но получает PostgreSQL-адаптер.

В тестах тот же use case можно создать с `InMemoryRequestRepository`. Это и есть смысл связывания в одном месте: application layer не знает, какой адаптер выбран.

### Вопрос 9. Почему Active Record нарушает Dependency Inversion Principle?

Active Record совмещает доменный объект и методы сохранения, например `request.save()`. Тогда сущность знает о БД или ORM, а значит домен зависит от инфраструктуры.

DIP требует обратного: инфраструктура должна зависеть от доменного контракта. В DDD доменный объект должен содержать бизнес-правила, а сохранение должен выполнять Repository Adapter.

### Вопрос 10. Как протестировать, что транзакция была зафиксирована (`committed`)?

Для unit-теста используют fake или in-memory Unit of Work с флагом `committed`. Use case выполняется успешно, вызывает `uow.commit()`, после чего тест проверяет `assert uow.committed is True`.

Отдельно нужно проверить негативный сценарий: если внутри use case выбрасывается исключение, `committed` должен остаться `False`, а `rolled_back` должен стать `True`. Так проверяется транзакционная дисциплина.

### Вопрос 11. Что такое Identity Map и когда он полезен?

Identity Map - паттерн, при котором в рамках одной транзакции или запроса один ID соответствует одному объекту в памяти. Если объект с таким ID уже загружен, повторная загрузка возвращает ту же ссылку.

Это полезно, чтобы не получить две разные копии одного агрегата и случайно не сохранить конфликтующие изменения. Также Identity Map уменьшает повторные обращения к БД внутри одного Unit of Work.

### Вопрос 12. Почему нельзя возвращать ORM-объекты из репозитория?

ORM-объекты являются инфраструктурными объектами. Они зависят от сессии, таблиц, lazy loading и схемы БД. Если репозиторий возвращает их наружу, домен и application layer начинают зависеть от ORM.

Repository должен возвращать доменные агрегаты. Маппинг из ORM/строки БД в доменный объект должен оставаться внутри adapter. Это сохраняет чистую границу и позволяет заменить ORM или БД.

### Вопрос 13. Как добавить второй репозиторий (`GroupRepositoryPort`) в Unit of Work?

Нужно определить новый порт `GroupRepositoryPort` в домене, реализовать его в infrastructure, например `PostgresGroupRepository` и `InMemoryGroupRepository`. Затем добавить свойство `groups` в `UnitOfWorkPort`.

Конкретный `PostgresUnitOfWork` будет создавать `self.groups = PostgresGroupRepository(connection)` вместе с `self.requests`. Use case сможет работать с `uow.requests` и `uow.groups`, оставаясь зависимым от абстракций.

### Вопрос 14. Почему «один агрегат - одна транзакция» (правило из лекции 07) и Unit of Work не противоречат друг другу?

Они отвечают за разные уровни. Правило агрегатов говорит о моделировании: не стоит без необходимости менять несколько агрегатов в одной бизнес-транзакции, потому что это увеличивает связанность и сложность.

Unit of Work - технический паттерн управления транзакцией use case. Он может содержать репозитории и commit/rollback, но хороший дизайн всё равно должен уважать границы агрегатов. Если use case затрагивает несколько агрегатов, нужно осознанно решить: это действительно одна транзакция или лучше использовать доменные события и eventual consistency.
